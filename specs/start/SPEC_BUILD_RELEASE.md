## 2. Dependency Strategy

### 2.1 tetra3

tetra3 is vendored at `python/vendor/tetra3/` and wired as an editable local dependency
in `pyproject.toml` via `[tool.uv.sources]`:

```toml
[tool.uv.sources]
tetra3 = { path = "python/vendor/tetra3", editable = true }
```

### 2.2 Python dependency locking

Uses `pyproject.toml` (hatchling build backend) with `uv` for dependency locking (`uv.lock`).

Dependencies: dearpygui, Pillow, numpy, scipy, tetra3, playsound3, aiohttp, qrcode[pil], pyerfa (>=2.0.0, for LX200 J2000‚ÜîJNow precession)
Dev dependencies: nuitka (>=4.0.2), pytest

## 3. macOS Build

### 3.1 Build the camera subprocess (Swift)

Uses Swift Package Manager (SwiftPM):

```bash
scripts/build_camera_mac.sh
# Runs: swift build -c release
# Produces: camera/mac/.build/release/camera_server
```

Backend uses:
- AVFoundation (capture)
- IOKit (UVC controls via USB control requests)

### 3.2 Build the Python app as a binary

Uses Nuitka in `--standalone` mode (produces a `.dist/` directory):

```bash
scripts/build_mac.sh
```

Key Nuitka flags:
- `--standalone` (NOT `--onefile` ‚Äî `.dist/` directory approach)
- `--output-filename=evf` (binary name; macOS/Linux) or `evf.exe` (Windows)
- `--include-package=erfa` (bundles the pyerfa C extension used by the LX200
  J2000‚ÜîJNow precession helpers)
- `--include-package=` for `dearpygui`, `numpy`, `scipy`, `PIL`, `playsound3`, `tetra3`
- Data files are NOT bundled via Nuitka ‚Äî copied manually post-build

## 4. macOS Packaging (.app and .dmg)

Final distributable: DMG containing `PushNav.app`

Expected `.app` layout:

```
PushNav.app/
  Contents/
    MacOS/
      evf                # Nuitka standalone dist (CFBundleExecutable)
      camera_server      # Swift binary
      ...                # Nuitka support files
    Resources/
      hip8_database.npz
      VERSION.json
      fonts/
      sounds/
      marketing/
    Info.plist
```

The Python app discovers resource paths relative to the executable location
via `evf/paths.py` (walks up to `../Resources` inside the .app).

Info.plist is generated by the build script with proper app metadata.

## 5. Linux Build

### 5.1 Camera server

```bash
make -C camera/linux
# Produces: camera/linux/camera_server
```

### 5.2 Python app

```bash
scripts/build_linux.sh
```

Uses Nuitka `--standalone`. Post-build assembles an AppImage:
- Downloads `appimagetool`
- Creates AppDir with Nuitka dist, camera binary, data files
- Includes `linux/pushnav.desktop` for desktop integration
- Produces both `PushNav-linux-{ARCH}.tar.gz` (raw standalone dist) and
  `PushNav-linux-{ARCH}.AppImage` (self-contained single-file distributable)

---

## 6. Windows Build

### 6.1 Camera server

```bash
camera\windows\build.bat
# Compiles C/DirectShow camera server
```

### 6.2 Python app + installer

```bash
scripts\build_windows.bat
```

Uses Nuitka `--standalone`. Post-build:
- Copies camera binary, data files, marketing assets
- Produces `PushNav-windows-x64.zip` (raw standalone dist, always)
- If Inno Setup is installed (`ISCC.exe` on PATH), also runs
  `scripts/pushnav.iss` to build `PushNav-windows-x64-setup.exe`. The
  installer step is skipped silently when Inno Setup is not present.

DPI awareness is handled at runtime (not build-time) via Windows API calls in `main.py`.

## 7. Config + Logs Paths

Config and logs are stored in platform-specific locations:

| Platform | Config root |
|----------|------------|
| macOS    | `~/Library/Application Support/ElectronicViewfinder/` |
| Linux    | `$XDG_CONFIG_HOME/electronic-viewfinder/` |
| Windows  | `%APPDATA%/ElectronicViewfinder/` |

Files:
- `config.json`
- `logs/evf.log` (rotating)

The app creates directories if missing.

## 8. Path Resolution (`evf/paths.py`)

Centralized path resolution handles dev repo, macOS .app bundle, Linux release, and
Windows release:

- `database_path()` ‚Äî star database
- `camera_binary_path()` ‚Äî platform-specific camera server binary
- `sounds_dir()` ‚Äî audio files
- `fonts_dir()` ‚Äî font files
- `title_image()` ‚Äî in-app branding

Detection heuristics:
- macOS release: `_MAC_RELEASE` ‚Äî executable inside `.app/Contents/MacOS/`
- Linux release: `_LINUX_RELEASE` ‚Äî `data/` sibling to executable
- Windows release: `_WINDOWS_RELEASE` ‚Äî `data/` sibling to executable
- Dev mode: falls back to repo-relative paths

## 9. Signing / Notarization

macOS commercial distribution requires:
- Code signing (`codesign`)
- Notarization (`xcrun notarytool`)
- Stapling (`xcrun stapler`)

These steps should be scripted in `scripts/build_mac.sh` once Apple Developer credentials are available.

## 10. Dev Mode (Fast Iteration)

Platform-specific dev scripts:

```bash
scripts/run_dev.sh              # macOS
scripts/run_dev_linux.sh        # Linux
scripts/run_dev_windows.bat     # Windows
```

These run the Python app in the normal interpreter, spawning the camera server
and using local data files. Avoids Nuitka while iterating on behavior and UI.

`scripts/run_mock_camera.sh` launches a mock camera server for testing without hardware.

Launch with `--dev` flag for debug features.

## 11. Versioning

`data/VERSION.json`:

```json
{
  "app_version": "<x.y.z>",
  "protocol_version": 1,
  "hip_db_version": "hip8_mag8_21200stars"
}
```

Concrete values drift per release; `protocol_version` and `hip_db_version` are
stable across minor releases while `app_version` tracks `pyproject.toml`.
Logged at startup (INFO).
